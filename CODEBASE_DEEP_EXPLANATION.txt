 1. PROJECT ARCHITECTURE OVERVIEW
================================================================================

This project follows Clean Architecture principles with four distinct layers:

  ┌─────────────────────────────────────────────────────┐
  │                  cmd/app/main.go                    │  ← Entry Point
  │          (Composition Root / Dependency Wiring)     │
  └─────────────────────┬───────────────────────────────┘
                        │ creates & injects
  ┌─────────────────────▼───────────────────────────────┐
  │              internal/tui/                           │  ← Presentation Layer
  │    AppModel, Screens, Components, Styles             │
  │    (BubbleTea framework, renders UI, handles input)  │
  └─────────────────────┬───────────────────────────────┘
                        │ calls services
  ┌─────────────────────▼───────────────────────────────┐
  │           internal/application/                      │  ← Application Layer
  │    BattleService, ShopService, ItemService            │
  │    (Orchestrates game logic, enforces rules)          │
  └─────────────────────┬───────────────────────────────┘
                        │ mutates/reads domain objects
  ┌─────────────────────▼───────────────────────────────┐
  │             internal/domain/                         │  ← Domain Layer
  │    Player, Enemy, Weapon, Inventory, Item, Element    │
  │    (Pure business entities, no external deps)         │
  └─────────────────────────────────────────────────────┘
  ┌─────────────────────────────────────────────────────┐
  │        internal/infrastructure/registry/             │  ← Infrastructure
  │    MonsterRegistry, WeaponRegistry, ItemRegistry      │
  │    (Static data stores, analogous to a database)      │
  └─────────────────────────────────────────────────────┘

DEPENDENCY DIRECTION: Outer layers depend on inner layers, never the reverse.
The domain layer has zero external dependencies.

 3. LAYER-BY-LAYER MODULE BREAKDOWN
================================================================================

--- DOMAIN LAYER (internal/domain/) ---

  Responsibility: Define pure business entities and rules.
  Dependencies:   NONE (zero imports from other internal packages)
  Principle:      Domain objects are the innermost layer. They know nothing
                  about the database, UI, or application orchestration.

  Key types:
  - Player:       Stats (HP, ATK, DEF, Gold), equipment, inventory, buffs
  - Enemy:        Stats, element, monster type (Normal/Elite/Boss)
  - Weapon:       ID, name, element, base damage, tier (upgradeable)
  - Item:         ID, name, category (Consumable/Material), effect type
  - Inventory:    Fixed-size slot array, methods to add/remove/query
  - Element:      Fire/Water/Earth/Wind with damage modifier matrix

--- APPLICATION LAYER (internal/application/) ---

  Responsibility: Orchestrate game use cases using domain objects.
  Dependencies:   domain, infrastructure/registry
  Principle:      Services contain game rules that span multiple domain
                  objects. They do NOT know about the UI.

  Key services:
  - BattleService:  Turn management, damage calculation, victory/defeat
  - ShopService:    Buy/sell transactions with gold validation
  - ItemService:    Centralized consumable effect processor (NEW)
                    Single source of truth for item effects.
                    Both BattleService and InventoryScreen delegate here.

--- INFRASTRUCTURE LAYER (internal/infrastructure/registry/) ---

  Responsibility: Provide static game data (analogous to a database).
  Dependencies:   domain
  Principle:      Data definitions are separate from logic.

  Registries:
  - MonsterRegistry:  All monster templates by ID and type
  - WeaponRegistry:   All weapon templates by ID
  - ItemRegistry:     All item templates by ID
  - UpgradeRegistry:  Upgrade cost tables per tier

--- PRESENTATION LAYER (internal/tui/) ---

  Responsibility: Render the terminal UI and handle user input.
  Dependencies:   application, domain, infrastructure, BubbleTea
  Principle:      The UI layer is the outermost layer. It depends on
                  everything but nothing depends on it.

  Layout: 3-panel design
    Block 1 (Left):    Menu component — navigation options
    Block 2 (Left):    Character component — player stats display
    Block 3 (Right):   Active screen — battle/shop/inventory/etc.

 4. DETAILED FILE & FUNCTION REFERENCE
================================================================================

--- cmd/app/main.go ---

  main()
    Entry point. Initializes registries, creates domain objects, wires
    application services with dependency injection, creates the TUI model,
    and starts the BubbleTea program with AltScreen mode.

  giveStartingItems(player, items, weapons)
    Provides initial equipment: 3x Small Potion + Iron Sword (equipped).
    Matches the original C version's starting loadout.

--- internal/application/item_service.go ---

  ItemService struct
    Holds a reference to the player. Processes consumable effects.

  NewItemService(player) → *ItemService
    Constructor. Takes the shared player pointer.

  UseConsumable(item) → UseResult
    Top-level dispatcher. Routes to specific effect handlers based on
    item.ConsumableType. Returns structured UseResult with Success/Message/Error.

  applyHeal(item) → UseResult
    Validates HP is not already full. Calls player.Heal() (domain method).
    Calculates actual HP recovered (may be less than item.Value due to cap).

  applyFullRestore(item) → UseResult
    Validates HP is not already full. Sets HP to MaxHP directly.

  applyBuff(item, buffType, label) → UseResult
    Calls player.AddBuff() to add a temporary stat modifier.
    No precondition — buffs can always be applied.

--- internal/application/battle_service.go ---

  BattleService struct
    Manages combat state machine (BattleNone → PlayerTurn → EnemyTurn → etc.)
    Holds references to player, current enemy, registries, and ItemService.

  NewBattleService(player, monsters, items, itemService) → *BattleService
    Constructor. Accepts the shared ItemService for item effect delegation.

  StartBattle(monsterType) → bool
    Selects a random monster from the registry. Initializes battle state.
    Returns false if a battle is already in progress.

  Attack()
    Calculates player damage (base + weapon + buffs + element modifier).
    Applies damage to enemy. If enemy survives, triggers enemy turn.
    If enemy dies, transitions to BattleWon state and awards gold.

  UseItem(slotIndex) → bool
    Validates slot contains a consumable. Delegates effect to ItemService.
    If ItemService rejects (e.g., HP full), logs error and returns false
    WITHOUT consuming the item or ending the player's turn.
    If successful, removes item and triggers enemy turn.

  TryEscape()
    30% base escape chance. Fails = enemy gets a free turn.
    Success = battle ends with BattleEscaped state.

  enemyTurn()
    Enemy attacks player. Checks for player death → BattleLost.

  EndBattle()
    Resets battle state to BattleNone. Ticks player buffs.

  GetLog() → []string
    Returns battle log entries. Capped at battleLogCapacity (20).

--- internal/application/shop_service.go ---

  ShopService struct
    Handles buy/sell transactions.

  BuyWeapon(weaponID) → error
    Validates gold. Creates weapon from registry. Adds to inventory.

  BuyItem(itemID, qty) → error
    Validates gold. Creates item from registry. Adds to inventory.

  SellItem(slotIndex) → error
    Validates slot. Calculates sell price. Removes from inventory.

--- internal/domain/player.go ---

  Player struct
    Core player entity. Fields: Name, MaxHP, CurrentHP, BaseAttack,
    BaseDef, Gold, EquippedWeapon, Inventory, Buffs.

  NewPlayer(name, config) → *Player
    Factory with configurable starting stats via PlayerConfig.

  IsAlive() → bool
    Returns true if CurrentHP > 0.

  GetAttack() → int
    Sums BaseAttack + weapon damage + percentage attack buffs.

  GetDefense() → int
    Sums BaseDef + percentage defense buffs.

  TakeDamage(amount) → bool
    Applies damage after defense reduction (50% of defense).
    Clamps HP to 0 minimum. Returns true if player died.

  Heal(amount)
    Increases CurrentHP by amount, capped at MaxHP.

  AddBuff(type, value, duration)
    Appends a Buff to the player's active buff list.

  TickBuffs()
    Decrements all buff durations. Removes expired buffs.
    Called at end of each battle via EndBattle().

--- internal/domain/inventory.go ---

  Inventory struct
    Fixed-size array of InventorySlot pointers.
    MaxInventorySlots = 20 (matches C version).

  InventorySlot struct
    Type (SlotWeapon/SlotItem), Weapon pointer, Item pointer, Quantity.

  AddWeapon(weapon) → error
    Finds empty slot, places weapon there. Error if inventory full.

  AddItem(item, qty) → error
    If item is stackable, adds to existing stack. Otherwise finds empty slot.

  RemoveItem(slotIndex, qty)
    Reduces quantity at slot. Removes slot if quantity reaches 0.

  GetSlot(index) → *InventorySlot
    Bounds-checked access to individual slots.

  GetWeapons() → []*InventorySlot
    Filters slots to return only weapon-type entries.

  GetConsumables() → []*InventorySlot
    Filters slots to return only consumable items (not materials).

  GetMaterials() → []*InventorySlot
    Filters slots to return only material items.

--- internal/domain/element.go ---

  Element type (int enum)
    ElementNone, ElementFire, ElementWater, ElementEarth, ElementWind

  DamageModifier(attacker, defender) → float64
    Returns damage multiplier based on elemental advantage:
    Fire > Wind > Earth > Water > Fire (1.5x advantage, 0.75x disadvantage)

--- internal/domain/errors.go ---

  ResultCode type (int enum)
    ResultSuccess, ResultFail, ResultInventoryFull, ResultNotEnoughGold, etc.

  DomainError struct
    Wraps ResultCode + optional message. Implements error interface.

--- internal/tui/model.go ---

  AppModel struct
    The root BubbleTea model. Holds:
    - All services (battle, shop, item)
    - All registries (weapons, items, monsters, upgrades)
    - UI components (menu, character)
    - Active screen (Block 3)
    - Layout state (width, height, focusedPanel)

  NewAppModel(...) → AppModel
    Constructor with full dependency injection.

  Init() → tea.Cmd
    BubbleTea lifecycle. Returns nil (no initial commands).

--- internal/tui/update.go ---

  AppModel.Update(msg) → (tea.Model, tea.Cmd)
    Central message router. Handles:
    - tea.WindowSizeMsg: Resizes all components
    - screens.EndScreenMsg: Returns from active screen to menu
    - tea.KeyMsg: Routes to focused panel (menu or game screen)
    - Ctrl+C / 'q': Quits application

  handleMenuSelect(id) → tea.Cmd
    Maps menu item IDs to screen constructors:
    - MenuBattleNormal/Elite/Boss → BattleScreen
    - MenuShop → ShopScreen
    - MenuInventory → InventoryScreen (with ItemService)
    - MenuEquip → EquipScreen
    - MenuUpgrade → UpgradeScreen
    - MenuStatus → StatusScreen

  focusGame()
    Sets focusedPanel to PanelGame and updates component focus states.

--- internal/tui/view.go ---

  AppModel.View() → string
    Assembles the 3-panel layout:
    1. Menu component (Block 1)
    2. Character component (Block 2)
    3. Active screen or welcome message (Block 3)
    Joins horizontally with proper sizing ratios.

  wrapGamePanel(content) → string
    Wraps raw content in a styled panel border with title.

--- internal/tui/screens/msg.go ---

  EndScreenMsg struct
    Empty message type. When received by AppModel.Update(), triggers
    return to main menu (activeScreen = nil, focusedPanel = PanelMenu).

  EndScreenCmd() → tea.Cmd
    Returns a tea.Cmd that sends EndScreenMsg. This is the ONLY correct
    way for a screen to signal "I'm done, return to menu."

--- internal/tui/screens/battle.go ---

  BattleScreen struct
    Mode (Main/ItemSelect/Result), cursor, battle service, player ref.

  Update(msg) → tea.Cmd
    Routes input by mode:
    - BattleModeMain: Attack/Items/Run selection
    - BattleModeItemSelect: Consumable picker
    - BattleModeResult: Enter/Esc exits battle via EndScreenCmd()

  handleSelect() → tea.Cmd
    Dispatches action based on cursor position in main mode.

  handleUseItem()
    Finds Nth consumable slot index. Calls service.UseItem(slotIndex).

  checkBattleState()
    Reads service state. If Won/Lost/Escaped, switches to BattleModeResult.

--- internal/tui/screens/inventory.go ---

  InventoryScreen struct
    Mode (Main/Weapons/Consumables/Materials/Use), cursor, player,
    itemService, message, msgStyle.

  Update(msg) → tea.Cmd
    Routes input. Esc from main mode sends EndScreenCmd().
    Esc from sub-modes returns to main mode.

  handleSelect() → tea.Cmd
    Routes main menu options (view weapons/consumables/materials/use/back).
    "Back" sends EndScreenCmd().

  handleUseItem()
    Gets consumable list. Delegates to itemService.UseConsumable().
    Maps UseResult to UI message/style. Removes item only on success.

  findConsumableSlotIndex(n) → int
    Maps filtered consumable index to raw inventory slot index.

 5. DATA FLOW EXPLANATION
================================================================================

ITEM USAGE FLOW (Battle context):

  1. Player presses "2" (Use Item) in battle → BattleScreen.handleSelect()
  2. Mode switches to BattleModeItemSelect
  3. Player selects a consumable → BattleScreen.handleUseItem()
  4. handleUseItem finds slot index → calls BattleService.UseItem(slotIdx)
  5. BattleService validates: is it player's turn? is slot a consumable?
  6. BattleService delegates: itemService.UseConsumable(item)
  7. ItemService validates: is HP full? is category correct?
  8. ItemService applies effect: player.Heal() or player.AddBuff()
  9. ItemService returns UseResult{Success:true, Message:"Recovered 30 HP"}
  10. BattleService logs message, removes item, ends turn, triggers enemy
  11. BattleScreen detects mode change, re-renders

ITEM USAGE FLOW (Inventory context):

  1. Player opens Inventory → selects "4. Use Item"
  2. InventoryScreen.handleSelect() sets mode to InventoryModeUse
  3. Player selects a consumable → InventoryScreen.handleUseItem()
  4. handleUseItem calls:  s.itemService.UseConsumable(item)
  5. ItemService validates and applies (SAME code path as battle)
  6. If success: removes item by slot index, shows green message
  7. If failure: shows red error message, does NOT remove item

 6. STATE MANAGEMENT EXPLANATION
================================================================================

NAVIGATION STATE:

  The AppModel manages a simple state machine with two variables:
  - focusedPanel (int): Which panel receives input (PanelMenu or PanelGame)
  - activeScreen (screens.Screen): The current game screen, or nil

  State transitions:
  - Menu → Screen:  handleMenuSelect() sets activeScreen, calls focusGame()
  - Screen → Menu:  Screen returns EndScreenCmd() → AppModel handles
                     EndScreenMsg by setting activeScreen=nil, focus=PanelMenu

BATTLE STATE:

  BattleService.state (BattleState enum):
    BattleNone → BattlePlayerTurn → BattleEnemyTurn → (loop)
                                  └→ BattleWon / BattleLost / BattleEscaped

  StartBattle() sets BattlePlayerTurn.
  Player actions transition to BattleEnemyTurn.
  enemyTurn() either loops back to PlayerTurn or ends battle.
  EndBattle() resets to BattleNone.

 7. CONSUMABLE ITEM LOGIC — COMPLETE PIPELINE
================================================================================

ITEM DEFINITION (domain/inventory.go):

  Item struct has:
  - ConsumableType: ConsumeHeal / ConsumeBuffAtk / ConsumeBuffDef / ConsumeFullRestore
  - Value: Effect magnitude (HP amount or buff percentage)
  - Duration: For buffs, number of turns active

ITEM REGISTRY (infrastructure/registry/items.go):

  Defines all item templates with IDs like "small_potion", "attack_scroll".
  Templates are cloned via ToItem() when added to inventory.

CENTRALIZED EFFECT PROCESSING (application/item_service.go):

  UseConsumable(item) validates preconditions and applies effects:

  ConsumeHeal:
    - Rejects if player.CurrentHP >= player.MaxHP ("HP is already full!")
    - Calls player.Heal(item.Value) which caps at MaxHP
    - Returns actual HP recovered (may be less than Value)

  ConsumeFullRestore:
    - Rejects if HP already full
    - Sets CurrentHP = MaxHP directly

  ConsumeBuffAtk / ConsumeBuffDef:
    - No precondition (buffs always stack)
    - Calls player.AddBuff(type, value, duration)
    - Buff expires when TickBuffs() is called (at battle end)

INVENTORY REMOVAL:

  ItemService does NOT remove items. Callers handle removal:
  - BattleService:     player.Inventory.RemoveItem(slotIndex, 1)
  - InventoryScreen:   finds slot index via findConsumableSlotIndex(), then
                        player.Inventory.RemoveItem(slotIndex, 1)

 8. NAVIGATION & SCREEN LIFECYCLE
================================================================================

SCREEN INTERFACE (screens/interface.go):

  Every screen implements:
  - Update(msg) → tea.Cmd    Handle input, return commands
  - View() → string          Render screen content
  - Title() → string         Panel title
  - SetSize(w, h)            Respond to terminal resize

SCREEN LIFECYCLE:

  1. CREATION:   NewXxxScreen(...) called in handleMenuSelect()
  2. ACTIVATION: activeScreen = screen; focusGame()
  3. INPUT:      AppModel.Update() delegates KeyMsg to activeScreen.Update()
  4. RENDER:     AppModel.View() calls activeScreen.View()
  5. EXIT:       Screen returns EndScreenCmd() → AppModel handles EndScreenMsg
  6. CLEANUP:    activeScreen = nil, focusedPanel = PanelMenu

CRITICAL INVARIANT:
  Every screen MUST have a code path that returns EndScreenCmd().
  Without this, the screen becomes a dead-end (cannot return to menu).
  This was the root cause of the original navigation bugs.

 9. FUTURE EXTENSIBILITY NOTES
================================================================================

ADDING A NEW CONSUMABLE TYPE:

  1. Add constant to domain/inventory.go ConsumableType enum
  2. Add case to ItemService.UseConsumable() switch
  3. Add item templates to registry/items.go
  4. No changes needed in BattleService or InventoryScreen

ADDING A NEW SCREEN:

  1. Create screens/xxx.go implementing Screen interface
  2. Add menu item ID to components/constants.go
  3. Add case to handleMenuSelect() in update.go
  4. MUST include EndScreenCmd() return path

ADDING A NEW SERVICE:

  1. Create application/xxx_service.go
  2. Add field to AppModel struct in model.go
  3. Accept in NewAppModel() constructor
  4. Wire in cmd/app/main.go

SAVE/LOAD SYSTEM (future):

  The current architecture makes save/load straightforward:
  - Serialize domain.Player (all game state is contained here)
  - Registries are static data (no need to save)
  - BattleService state is transient (no mid-battle saves needed)
