1. EXECUTIVE SUMMARY
================================================================================

TextRPG is a terminal-based role-playing game built in Go using the Charm
BubbleTea framework for terminal user interface (TUI) rendering. The project
follows Clean Architecture principles with clear separation of concerns across
four architectural layers:

  - Domain Layer:          Pure business logic, no external dependencies.
  - Application Layer:     Service orchestration (battle, shop, item usage).
  - Infrastructure Layer:  Data registries (monsters, weapons, items, upgrades).
  - Presentation Layer:    BubbleTea TUI with screens, components, and styles.

The game features:
  • Turn-based combat with elemental damage modifiers (Fire > Wind > Earth > Water > Fire)
  • Weapon system with tier-based upgrades and elemental affinities
  • Shop system for buying/selling weapons and consumables
  • Inventory management with stacking, filtering, and slot-based storage
  • Consumable items (healing potions, buff elixirs) with centralized effect processing
  • Monster hierarchy (Normal, Elite, Boss) with loot tables
  • Material/crafting system for weapon upgrades
  • 3-panel TUI layout (Menu | Character Status | Game Screen)

The codebase is a port of an original C implementation and maintains feature
parity with the C version's data layer (items, monsters, weapons, upgrades).

3. ARCHITECTURE OVERVIEW
================================================================================

The project follows the Clean Architecture pattern (a.k.a. Onion Architecture),
where dependencies flow inward: outer layers depend on inner layers, but never
the reverse.

                    ┌───────────────────────────────────────┐
                    │           TUI (Presentation)          │
                    │   model.go, view.go, update.go        │
                    │   screens/*, components/*, styles/*    │
                    ├───────────────────────────────────────┤
                    │        Application (Use Cases)         │
                    │   battle_service.go                    │
                    │   item_service.go                      │
                    │   shop_service.go                      │
                    ├───────────────────────────────────────┤
                    │       Infrastructure (Data)            │
                    │   registry/items.go                    │
                    │   registry/monsters.go                 │
                    │   registry/upgrades.go                 │
                    │   registry/weapons.go                  │
                    ├───────────────────────────────────────┤
                    │         Domain (Core Logic)            │
                    │   element.go, enemy.go, player.go      │
                    │   weapon.go, inventory.go, errors.go   │
                    └───────────────────────────────────────┘

DEPENDENCY RULES:
  • Domain depends on NOTHING external (pure Go, no imports from other layers)
  • Infrastructure depends on Domain (uses domain types like Element, ItemCategory)
  • Application depends on Domain + Infrastructure (orchestrates business actions)
  • TUI depends on ALL layers (renders UI using services and domain entities)

LAYER RESPONSIBILITIES:
  • Domain:         Defines WHAT entities exist and HOW they behave in isolation.
  • Infrastructure: Defines WHERE data comes from (in-memory registries).
  • Application:    Defines HOW multi-entity workflows operate (battle turns, shop).
  • TUI:            Defines HOW the user interacts with the system.

================================================================================
5. ENTRY POINT & EXECUTION FLOW
================================================================================

STARTUP SEQUENCE (main.go):
  1. Create all four registries (monsters, weapons, items, upgrades).
     These are read-only data stores populated at initialization.
  2. Create a new Player with default configuration (100 HP, 300 Gold, 10 DEF).
  3. Give starting equipment: 3x Small Potion + Iron Sword (equipped).
  4. Create application services (ItemService, BattleService, ShopService).
     ItemService is shared between BattleService and InventoryScreen.
  5. Create AppModel (root TUI model) injecting all dependencies.
  6. Start BubbleTea program with alternate screen mode.

RUNTIME LOOP (BubbleTea Elm Architecture):
  The application follows the Elm Architecture pattern:

    Update(msg) → tea.Model, tea.Cmd
    View()      → string

  1. User presses a key → tea.KeyMsg is dispatched.
  2. AppModel.Update() routes the message:
     a. If menu is focused → menu.Update() handles navigation.
     b. If Enter/number pressed → handleMenuSelect() creates appropriate screen.
     c. If game screen focused → activeScreen.Update() handles screen-specific input.
  3. AppModel.View() renders the 3-panel layout:
     a. Block 1 (Left):   Menu component → main navigation.
     b. Block 2 (Center): Character component → player status, HP bar, weapon.
     c. Block 3 (Right):  Active screen → battle/shop/inventory/etc.
  4. When a screen finishes → EndScreenMsg is sent → focus returns to menu.

SHUTDOWN:
  Exit via Ctrl+C or selecting "Exit" from main menu → tea.Quit command.

6.2 DOMAIN LAYER (internal/domain/)
--------------------------------------------------------------------------------

FILE: element.go
PURPOSE: Elemental type system and damage modification.
LINES: 86
DEPENDENCIES: None (pure domain logic)

  Defines the Element type (an int enum) with four values: Fire, Water, Earth,
  Wind, plus None for non-elemental entities.

  Implements a rock-paper-scissors damage modifier system:
    Fire  → strong vs Wind,  weak vs Water
    Water → strong vs Fire,  weak vs Earth
    Earth → strong vs Water, weak vs Wind
    Wind  → strong vs Earth, weak vs Fire

  Named constants prevent magic numbers:
    ElementStrongMultiplier  = 2.0 (double damage on advantage)
    ElementWeakMultiplier    = 0.5 (half damage on disadvantage)
    ElementNeutralMultiplier = 1.0 (no elemental interaction)

  The DamageModifier(target Element) method computes the multiplier when an
  entity with this element attacks a target with the given element.

FILE: enemy.go
PURPOSE: Enemy entity definition and combat behavior.
LINES: 76
DEPENDENCIES: math/rand (for gold rolls), domain constants

  Defines the Enemy struct with ID, Name, Type, Element, HP, Attack, Defense,
  and gold range.

  MonsterType enum categorizes enemies for difficulty scaling:
    MonsterNormal → standard encounters
    MonsterElite  → increased stats and better loot
    MonsterBoss   → cannot be fled, highest rewards

  Key methods:
    IsAlive()     → checks if CurrentHP > 0
    TakeDamage()  → applies damage with defense reduction (DEF * 0.5)
                     enforces minimum damage of 1, returns true if killed
    RollGold()    → random gold in [GoldMin, GoldMax] range
    GetAttack()   → returns raw attack value
    GetElement()  → returns elemental affinity

FILE: errors.go
PURPOSE: Domain-specific error types and result codes.

DEPENDENCIES: None

  DomainError struct wraps a ResultCode with an optional message and

    Item          → consumable or material definition

FILE: player.go
PURPOSE: Player entity with stats, equipment, buffs, and combat.
LINES: 171
DEPENDENCIES: domain constants

  The central entity of the game. Holds all mutable player state:
    Name, MaxHP, CurrentHP, BaseAttack, BaseDef, Gold,
    EquippedWeapon (*Weapon), Inventory (*Inventory), Buffs ([]Buff)

  BUFF SYSTEM:
    Buff struct holds Type (BuffAttack/BuffDefense), Value (percentage),
    and TurnsRemaining. Buffs are applied multiplicatively to stats.

  COMBAT BALANCE CONSTANTS:
    DefenseReductionRatio = 0.5 (defense reduces damage by 50% of DEF value)
    DefaultBaseAttack = 10
    MinDamage = 1 (prevents zero-damage hits)

  CONSTRUCTOR:
    NewPlayer(name, config) creates player with configurable starting stats.
    DefaultPlayerConfig() returns: 100 HP, 300 Gold, 10 DEF.

  KEY METHODS:
    GetAttack()   → BaseAttack + weapon damage + buff bonuses (percentage)
    GetDefense()  → BaseDef + buff bonuses (percentage)
    TakeDamage()  → reduces HP by (amount - DEF*0.5), min 1 damage
    Heal()        → restores HP capped at MaxHP
    SpendGold()   → deducts gold, returns error if insufficient
    AddBuff()     → appends temporary buff
    TickBuffs()   → decrements durations, removes expired (uses slice reuse pattern)

FILE: weapon.go
PURPOSE: Weapon entity with type classification and tier upgrades.
LINES: 68
DEPENDENCIES: fmt (for display name formatting)

FILE: battle_service.go
PURPOSE: Combat orchestration, turn management, and victory/defeat handling.
LINES: 271
DEPENDENCIES: domain, infrastructure/registry, math/rand, fmt

  DESIGN PATTERN: Service (orchestrates multi-entity business operations)

  BattleState enum: None, PlayerTurn, EnemyTurn, Won, Lost, Escaped

  COMBAT BALANCE:
    CriticalHitChance     = 0.10 (10% crit rate)
    CriticalHitMultiplier = 1.5  (150% damage on crit)
    EscapeChance          = 0.5  (50% flee success, bosses cannot be fled)
    battleLogCapacity     = 20   (ring buffer for combat log)

  KEY METHODS:
    StartBattle()   → picks random monster of given type, initializes combat
    Attack()        → player attacks enemy with weapon damage + element modifier + crit roll
    UseItem()       → delegates to ItemService for effect, removes item on success
    TryEscape()     → 50% chance, blocked for bosses
    enemyTurn()     → enemy attacks player, ticks buffs after full round
    onEnemyDefeated() → grants gold reward, sets BattleWon state

  The UseItem() method demonstrates the centralized pattern: it validates the
  slot, delegates effect application to ItemService, and only removes the item
  and advances the turn if the effect succeeded.

FILE: item_service.go
PURPOSE: Centralized consumable effect processor (Single Source of Truth).
LINES: 163
DEPENDENCIES: domain, fmt

  DESIGN RATIONALE:
    This service was created to eliminate duplicated item-effect logic between
    BattleService and InventoryScreen. Previously, BattleService used
    player.Heal() while InventoryScreen directly mutated player.CurrentHP.
    Buff logic was entirely missing from the inventory path. This service
    unifies both paths.

  UseResult struct separates:
    Success → whether the effect was applied
    Message → human-readable outcome (for UI display)
    Error   → reason for failure (for UI display)

  UseConsumable() dispatches by ConsumableType:
    ConsumeHeal       → heals by item.Value, fails if HP full
    ConsumeFullRestore → sets HP to max, fails if HP full
    ConsumeBuffAtk    → adds attack buff for item.Duration turns
    ConsumeBuffDef    → adds defense buff for item.Duration turns

  IMPORTANT: This service does NOT remove items from inventory. The caller
  is responsible for removal, as different contexts use different removal
  strategies (slot-index in battle, consumable-index in inventory).

FILE: shop_service.go
PURPOSE: Buy/sell transaction management.
LINES: 128
DEPENDENCIES: domain, infrastructure/registry

  CONSTANTS:
    DefaultSellRatio = 0.5 (player receives 50% of base sell price)

  KEY METHODS:
    BuyWeapon()     → validates gold, creates weapon from template, adds to inventory
    BuyItem()       → validates gold, creates item from template, adds to inventory
    SellItem()      → handles both weapon and item selling from inventory slots
    GetShopWeapons() → delegates to WeaponRegistry for stock info
    GetShopItems()   → delegates to ItemRegistry for stock info

  NOTE: The sell ratio is applied only to weapon sell prices (calculated from
  template). Item sell prices use the direct SellPrice field from the item.

================================================================================
7. DESIGN PATTERNS & PRINCIPLES
================================================================================

1. CLEAN ARCHITECTURE (Onion/Hexagonal)
   Domain layer has ZERO external dependencies. All business rules live in
   pure Go with no framework coupling. The TUI layer can be replaced with
   a different UI framework without changing domain or application logic.

2. DEPENDENCY INJECTION
   All services and registries are constructed in main.go and injected
   through constructors. No global state, no singleton patterns.

3. REGISTRY PATTERN
   Data registries (items, weapons, monsters, upgrades) act as read-only
   data stores. They encapsulate data loading and provide typed query
   methods (GetByID, GetAll, GetByType, GetRandom).

4. ELM ARCHITECTURE (TEA — The Elm Architecture)
   BubbleTea implements Model-Update-View:
   - Model: AppModel holds all state
   - Update: processes messages, returns new model + commands
   - View: renders model state to string

5. COMMAND PATTERN
   BubbleTea commands (tea.Cmd) encapsulate side effects. EndScreenCmd()
   creates a command that emits EndScreenMsg when executed.

6. TEMPLATE PATTERN (Registry Templates → Domain Entities)
   MonsterTemplate.ToEnemy(), WeaponTemplate.ToWeapon(), ItemTemplate.ToItem()
   create mutable domain entities from immutable template data.

7. SINGLE SOURCE OF TRUTH (ItemService)
   Consumable effect logic is centralized in ItemService, preventing
   duplication between BattleService and InventoryScreen.

8. STRATEGY PATTERN (Screen Interface)
   The Screen interface allows AppModel to swap game screens without
   knowledge of concrete implementations.

9. NAMED CONSTANTS (No Magic Numbers)
   All numeric values have descriptive constant names:
   DefenseReductionRatio, CriticalHitChance, MaxInventorySlots, etc.

8. DATA FLOW DIAGRAMS
================================================================================

BATTLE FLOW:
  Menu "Battle (Normal)" →
    handleMenuSelect() →
      BattleService.StartBattle(MonsterNormal) →
        MonsterRegistry.GetRandom(Normal) →
          MonsterTemplate.ToEnemy() →
            BattleScreen created, focus shifts to game panel →
              Player selects "Attack" →
                BattleService.Attack() →
                  Player.GetAttack() + Weapon.Element.DamageModifier() →
                    Enemy.TakeDamage() →
                      [If killed] → BattleService.onEnemyDefeated() →
                        Enemy.RollGold() → Player.AddGold() →
                          BattleWon → BattleModeResult →
                            Enter pressed → EndScreenMsg → Menu

ITEM USAGE (Battle):
  BattleScreen "Use Item" →
    Player.Inventory.GetConsumables() →
      Select item → findConsumableSlotIndex() →
        BattleService.UseItem(slotIndex) →
          ItemService.UseConsumable(item) →
            [ConsumeHeal] → Player.Heal() → UseResult{Success=true}
            → RemoveItem(slotIndex, 1)
            → BattleEnemyTurn → enemyTurn()

ITEM USAGE (Inventory — same path):
  InventoryScreen "Use Item" →
    ItemService.UseConsumable(item) → [same effect logic]
      → findConsumableSlotIndex() → RemoveItem()

SHOP PURCHASE:
  ShopScreen "Buy Weapons" →
    WeaponRegistry.GetShopWeapons() →
      Select weapon → ShopService.BuyWeapon(weaponID) →
        WeaponTemplate → Player.Gold check → Inventory.AddWeapon()
        → Gold deducted → Success message


================================================================================
9. POTENTIAL IMPROVEMENTS & RECOMMENDATIONS
================================================================================

1. LOOT TABLE PROCESSING
   The BattleService.onEnemyDefeated() method has a TODO comment for loot
   drops. MonsterTemplate.LootTable is defined but never processed during
   combat. This should be implemented to generate item drops based on
   drop chances after defeating monsters.

2. SHOP STOCK DECREMENT
   WeaponRegistry and ItemRegistry have DecrementStock() methods, but
   ShopService.BuyWeapon() and BuyItem() do not call them. Limited-stock
   items are never actually decremented, making stock limits purely cosmetic.

3. ANTIDOTE CONSUMABLE TYPE
   The Antidote item has ConsumableType = ConsumeNone, meaning ItemService
   will reject its usage with "This item cannot be used." Either add a
   poison/status system or change the antidote's type.

4. INTERFACE ABSTRACTION FOR SERVICES
   BattleService, ShopService, and ItemService are concrete types injected
   directly. Defining interfaces for these services would improve testability
   and enable mock injection for unit tests.

5. WEAPON SELL PRICE INCONSISTENCY
   ShopService.SellItem() applies sellRatio (0.5) only to weapon sell prices
   from the template, but uses item.SellPrice directly for items without
   applying the ratio. This creates an inconsistent pricing model.

6. CURSOR CLAMPING DUPLICATION
   Every screen (Battle, Shop, Upgrade, Equip, Inventory) has its own
   clampCursor() method with identical logic. This should be extracted
   into a shared utility function.

7. BUFFER REUSE IN TICK BUFFS
   Player.TickBuffs() uses the "slice reuse" pattern (active := p.Buffs[:0])
   which is memory-efficient but may cause subtle bugs if any code holds
   references to the old Buffs slice, as the old entries are not zeroed.

8. ERROR HANDLING IN EQUIP SCREEN
   EquipScreen.handleEquip() calls AddWeapon() for the old weapon but does
   not check the return code. If inventory is full, the old weapon would be
   silently lost. This should be validated before removing the new weapon.

9. PERSISTENT STATE
   There is no save/load system. Game state is lost on exit. Implementing
   serialization (JSON/GOB) for Player and Inventory would enable saving.

10. SCREEN RESIZING ROBUSTNESS
    The SetSize type assertion in resizeComponents() uses an interface check
    that won't fail silently. However, the Screen interface already has
    SetSize() — the type assertion is redundant and should use the interface
    method directly.
